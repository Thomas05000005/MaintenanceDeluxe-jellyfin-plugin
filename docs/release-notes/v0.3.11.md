# 🚀 MaintenanceDeluxe v0.3.11 — Fix cache busting des assets admin

Cette release ferme **définitivement** une classe de bug "stale admin UI après upgrade plugin" qui frappait depuis v0.3.4 sans qu'on le remarque jusqu'à ce qu'elle gâche v0.3.10.

Le symptôme : tu update le plugin de v0.3.9 vers v0.3.10, tu ouvres la page Paramètres, et tu vois l'ancienne UI v0.3.9 avec un layout cassé. Le DLL est bien installé en v0.3.10, mais le browser sert encore `admin.js` et `admin.css` depuis son cache local (`Cache-Control: max-age=300` posé en v0.3.4 = 5 min de cache aveugle).

**À partir de v0.3.11, chaque release force un fetch frais des assets sans aucune intervention de l'admin.**

## ✨ Highlights

### 🐛 Le bug

Depuis v0.3.4, les 3 assets admin (`banner.js`, `admin.js`, `admin.css`) étaient servis avec :

```
Cache-Control: public, max-age=300
```

Le browser cache aveuglément 5 minutes — pratique pour les fetchs répétés dans la même session, mais **catastrophique après un upgrade plugin** : le browser ignore le nouveau code et continue de servir l'ancien jusqu'à ce que les 5 min expirent (ou que l'admin fasse `Ctrl+Shift+R`).

Cela a frappé v0.3.10 frontalement : le user a installé la version, ouvert la page admin, et n'a vu **aucun** des changements promis (live preview, layout grid, fieldsets). L'UI était l'ancienne v0.3.9.

### 🔧 Le fix — ETag tied to assembly version

Le helper `ServeEmbeddedAsset` est réécrit pour utiliser une stratégie **ETag-based revalidation** :

- **ETag** calculé une fois au startup à partir de `Assembly.GetName().Version` :
  ```csharp
  private static readonly string _assetEtagVersion = "\"v" + version + "\"";
  ```
- **Cache-Control** : `public, no-cache, must-revalidate`
  - ⚠️ `no-cache` n'est pas "ne pas cacher" — c'est "**doit revalider** avec ETag avant de réutiliser le cache". `no-store` serait le "ne pas cacher du tout"
- **Validation conditionnelle** :
  - Le browser envoie `If-None-Match: "v0.3.11.0"` à chaque request
  - Si le serveur match → retour **304 Not Modified** (zero body, très rapide)
  - Sinon → retour **200** avec le nouveau body + nouveau ETag

### 🎯 Conséquence pratique

| Scénario | Comportement |
|---|---|
| Admin ouvre la page paramètres pour la 1ʳᵉ fois après l'install v0.3.11 | GET 200 avec body, browser cache l'asset avec ETag `"v0.3.11.0"` |
| Admin navigue ailleurs et revient sur la page paramètres | GET 304, body vide, browser réutilise son cache (très rapide) |
| Admin upgrade vers v0.3.12 et ouvre paramètres | GET avec `If-None-Match: "v0.3.11.0"` → serveur compare à `"v0.3.12.0"` → mismatch → 200 avec nouveau code, ETag mis à jour |

**Plus jamais de stale admin UI après upgrade.** L'utilisateur n'a rien à faire (pas de hard-refresh nécessaire).

## 🔧 Détails techniques

- L'ETag est calculé **une seule fois** au startup du process Jellyfin via `typeof(BannerController).Assembly.GetName().Version`. C'est un `static readonly` → zero overhead par request
- Le retour 304 est `StatusCode(StatusCodes.Status304NotModified)` sans body, conforme RFC 7232. Le browser réutilise alors le cache local sans recharger les bytes
- L'ETag est posé sur la response **304 ET 200** pour cohérence — si une couche intermédiaire (reverse proxy) check l'ETag, elle aura toujours la valeur
- `X-Content-Type-Options: nosniff` reste posé sur le retour 200 uniquement (pas besoin sur 304 qui n'a pas de body à sniffer)
- 121 tests xUnit inchangés (la logique business n'est pas touchée)

## 📝 Notes

- **Si tu es bloqué sur v0.3.10 avec une UI cassée** : fais `Ctrl+Shift+R` sur la page admin pour forcer le refetch des assets v0.3.10. Une fois fait, l'éditeur Annonces s'affichera correctement (live preview + fieldsets + comparisons alignées)
- v0.3.11 est un **fix opérationnel pur** — pas de nouvelle feature, pas de changement de comportement runtime côté utilisateur final. C'est uniquement la stratégie de cache HTTP qui change
- À toutes les futures releases, ce problème ne se reproduira plus jamais : l'ETag bumpe automatiquement avec la version d'assembly, donc 0 risque de stale assets
