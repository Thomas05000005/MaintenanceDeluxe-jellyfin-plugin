# 🔍 MaintenanceDeluxe v0.7.0 — Audit secondaire (11 fixes + 32 tests)

Cette release est le résultat d'un **audit secondaire** ciblant les zones non couvertes par l'audit pré-v0.6.1 : code legacy, patterns de concurrence, webhooks, scheduled task, sessions, configuration XML. **Chaque fix a été vérifié avant ET après** (reproduction du bug, puis test du fix).

## 🛡️ Sécurité

### SSRF defense pour les webhook URLs

Avant v0.7.0, validation = juste `scheme === https`. Tu pouvais configurer un webhook vers :
- `https://127.0.0.1/internal-service` → le serveur Jellyfin appelle son propre réseau interne
- `https://169.254.169.254/latest/meta-data/iam/security-credentials/` → AWS metadata = exfiltration credentials
- `https://10.0.0.1/router-admin` → reach LAN privé depuis le serveur

Nouvelle helper `IsWebhookHostSafe(url)` qui rejette :
| Range | Exemple bloqué |
|---|---|
| Loopback IPv4 | `127.0.0.1`, `127.255.255.255` |
| Loopback IPv6 | `::1` |
| Link-local IPv4 | `169.254.0.0/16` (cloud metadata) |
| Link-local IPv6 | `fe80::/10` |
| RFC1918 privé | `10/8`, `172.16/12`, `192.168/16` |
| IPv6 ULA | `fc00::/7` |
| Hostnames | `localhost`, `*.localhost`, `*.internal`, `*.local` |
| Réservé | `0.0.0.0/8` |

Appliqué sur **`SaveMaintenance`** (au save de la config webhook) **et** **`TestWebhook`** (sinon admin pouvait tester avant save).

**Pas de DNS lookup** (latence + TOCTOU). Blocklist purement statique sur hostnames + IP literals.

### Log sanitization webhook

L'URL webhook contient un token (Discord/Slack). Avant, l'exception loggée pouvait contenir l'URL avec son token → leak dans les fichiers de log (souvent lus par des utilisateurs moins privilégiés).

Nouvelle helper `SanitiseExceptionMessage(message, url)` qui remplace l'URL + hostname par `[redacted-webhook-url]` / `[redacted-host]` avant logging.

`TestAsync` retourne maintenant un message générique au lieu d'`ex.Message` (qui pouvait leak des hostnames internes via SSL cert subject names type `CN=internal-api.local`).

## 🔴 Bugs critiques corrigés

### `checkTimeWindow` JS sans midnight wrap

**Reproduction du bug (avant fix)** :
```javascript
// Schedule daily 22:00 - 06:00, on est à 23h00
checkTimeWindow_BUGGY(date23h, '22:00', '06:00')  // → false ❌
checkTimeWindow_BUGGY(date03h, '22:00', '06:00')  // → false ❌
```

Le C# `MatchesTimeWindow` (dans `AnnouncementHelper`) gérait correctement le wrap depuis v0.5.2. Le JS, non. Résultat : bannières/annonces avec fenêtre overnight invisibles la nuit côté client.

**Après fix** : 15 cas testés (overnight, normal, boundary, open-ended) tous OK.

### `Announcements` / `AnnouncementsSeen` `NullReferenceException` potentielle

```csharp
// Avant :
public List<Announcement> Announcements { get; set; }       // pas d'initializer
public List<AnnouncementsSeenEntry> AnnouncementsSeen { get; set; }
```

Le constructeur les init bien, mais `XmlSerializer` peut bypass le constructeur sur XML legacy contenant `xsi:nil="true"`. Résultat : `config.Announcements.FirstOrDefault(...)` ligne 662 → NRE.

Fix : `= new()` au niveau property.

## 🟠 Bugs sérieux corrigés

### `ValidateSchedules` `WeekDays` empty/null — auto-correction

v0.6.1 rejetait au save : breaking change pour les configs legacy qui avaient des rotation messages `type: weekly, weekDays: []` (interprétés comme "never-active" en silence). v0.7.0 **auto-normalise en `type: "always"`** — backward-compat préservée.

### `DetectFormat` substring → host parsing

```csharp
// Avant : substring match (fragile)
if (url.Contains("discord.com/api/webhooks/", ...)) return Discord;
// → https://relay.com/discord-bot/api/webhooks/proxy classifié Discord à tort

// Après : Uri.Host exact match (correct)
if (host.Equals("discord.com", ...) ...) return Discord;
```

### `GetActiveSessions` race condition

`_sessionManager.Sessions.Where(...)` itère sur collection mutable. Snapshot via `.ToList()` avant les opérateurs LINQ.

### `ClonePublicFields` réutilisé par `MaintenanceScheduleTask`

`MaintenanceScheduleTask` avait son propre snapshot field-by-field qui **manquait** `ScheduledStart`, `ScheduledEnd`, `ActivatedAt`. Donc le webhook deactivation notification arrivait avec ces champs à null. Rendu `internal` et réutilisé.

### Webhook `Retry-After` respecté

Avant : 429 → retry immédiat → re-foire. Après : parse le header `Retry-After` (seconds ou HTTP-date), attend si ≤ 4s. Si > 4s, surface le 429 au caller au lieu d'un retry agressif.

### Webhook payload size cap 32KB

Évite Discord/Slack/proxy 400 sur payloads trop gros. Si `customTitle` + `customSubtitle` + `message` font > 32KB → log warning + retour `(0, "Payload too large…")`.

## 🟡 Hardening

### `banner.js` `tick()` defensive null check

```javascript
function tick() {
  if (!CONFIG) { hideBanner(); return; }  // v0.7.0
  if (CONFIG.showInDashboard === false && isAdminPage()) ...
}
```

Si `/config` fetch fail (plugin disabled mid-session, network blip, server restart), `CONFIG` stays `null` et chaque `CONFIG.*` throw. Early return = bannière disparaît au lieu de cracher.

### `LastModified` monotone

```csharp
// Avant :
config.LastModified = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
// Après :
var now = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
config.LastModified = Math.Max(config.LastModified, now);
```

Protège contre les horloges qui reculent (NTP correction, admin manuel) qui casseraient le polling `if-modified-since` côté client.

## ❌ Faux positifs identifiés

| Finding agent | Réalité |
|---|---|
| "SaveConfiguration hors du mutex" | **Faux** : déjà dans le `try { ... }` couvert par `_mutex.WaitAsync()` (3 méthodes: Activate, Deactivate, EnsureUsersDisabled) |
| "Custom theme exposé via /config = info leak" | **Faux** : ajouté volontairement en v0.6.0 pour cohérence avec `announcementTheme`, documenté |
| "Double-escaping dans mdToHtml" | **Faux** : le `&quot;` est correctement interprété dans `href` par le browser |
| "isDraft toujours read as false" | **Faux** : `(row.querySelector(...) || {}).checked` fait le bon fallback `false` quand le field n'existe pas (rows legacy) |
