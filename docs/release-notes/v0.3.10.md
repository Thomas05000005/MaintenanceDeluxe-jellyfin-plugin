# 🚀 MaintenanceDeluxe v0.3.10 — Refonte complète de l'éditeur Annonces

Cette release répond directement aux **trois problèmes critiques** remontés sur l'éditeur d'annonces livré en Phase 1 (v0.3.9) : un Save global qui ignorait les annonces, un layout désordonné, et l'absence de prévisualisation. Tout est repris à zéro avec une UI propre, alignée, structurée en **fieldsets thématiques** avec **live preview à droite**.

Pas de changement du runtime banner.js — le rendu côté utilisateurs finaux reste identique à v0.3.9. C'est uniquement l'admin UI qui change.

## ✨ Highlights

### 🛡️ Le bouton Enregistrer global save enfin les annonces

- **🐛 Le bug** : depuis v0.3.9, cliquer *Enregistrer* en bas de la page ne déclenchait que `POST /config` + `POST /maintenance`. Les annonces étaient **ignorées** sauf si on cliquait hors d'un champ pour trigger l'auto-save sur `change` — fragile et invisible
- **🔧 Le fix** : le handler global lance désormais **3 sauvegardes en parallèle** via `Promise.allSettled` :
  1. `POST /config` (bannière + rotation + preset)
  2. `POST /maintenance` (toggle, schedule, release notes)
  3. `POST /announcements/admin` (toutes les annonces + multi-mode)
- **📢 Rapport d'erreur unifié** : si une (ou plusieurs) save échoue, le message liste précisément lesquelles + le détail server-side. Plus de save silencieusement perdu

### 🎨 Layout grid 2 colonnes + fieldsets

Avant : tout était empilé en blocs, les comparaisons utilisaient des `is="emby-input"` avec un grid CSS qui se cassait au wrap.

Maintenant :

- **📐 Editor en grid `1fr 1fr`** : formulaire à gauche, **live preview à droite** (passe en 1 colonne sous 1100 px)
- **🗂️ 6 fieldsets thématiques** dans le form, chacun avec `<legend>` qui crée une visual hierarchy claire :
  - Identité (icon / titre / version)
  - Corps (markdown)
  - État & importance (niveau colorisé + active)
  - Cible (rôles + utilisateurs spécifiques)
  - Comparaisons avant / après
  - Bouton d'action (CTA)
- **🪶 Inputs alignés** : tous les champs utilisent la même classe `.jf-ann-field` avec label au-dessus, input full-width, hint en-dessous. Pas de mix `inputContainer` / div nu / `emby-input`
- **📊 Comparisons en grid 5 colonnes** : `Label | Before | After | Highlight | ×`. Sous 800 px, passe en 2 colonnes avec le delete en bout de seconde ligne

### 🪟 Live preview à droite

C'était la lacune la plus visible.

- **🎬 Rendu local de la modal** (pas d'iframe — la modal d'annonce ne dépend pas du contexte Jellyfin, donc HTML local suffit)
- **🌌 Background style** : radial-gradient or/midnight qui mime visuellement le fond aurora de la maintenance overlay
- **🎯 Accent colorisé selon l'importance** : la barre du haut de la modal + les liens dans le markdown utilisent la couleur du niveau choisi (bleu / vert / orange / rouge)
- **🔄 Mise à jour à chaque keystroke** : tape dans le titre, le body, ajoute une comparison, change l'importance — la preview reflète instantanément
- **📐 Sticky `top: 1em`** : le preview suit le scroll de la colonne form. Tu peux scroller dans les fieldsets sans perdre le rendu de vue

### 💾 Save explicite avec badge dirty pulsant

Au lieu de l'auto-save invisible :

- **🔘 Bouton "Enregistrer cette annonce"** dédié par row (vert, primary), à la base du fieldset
- **🟠 Badge dirty pulsant** : dès qu'un champ est modifié, le bouton passe en orange avec une animation `pulse` 2.4 s. Visible immédiatement, impossible à ignorer
- **✓ Feedback de succès** : après save réussi, le bouton affiche "✓ Enregistré" pendant 1.5 s puis revient au normal
- **❌ Feedback d'erreur** : `Dashboard.alert()` avec le détail server-side

### 🎨 174 lignes de CSS dédié

Nouveau bloc `Announcements editor` dans `admin.css` avec :

- **`.jf-ann-row`** : container collapsible, bordure dorée en mode expanded, transition douce
- **`.jf-ann-summary`** : barre header avec icon + title + ratio "vu par" + status badge (Active / Inactive) + chevron animé
- **`.jf-ann-editor`** : le grid 2-colonnes (form + preview)
- **`.jf-ann-form fieldset` + `legend`** : style typographique cohérent avec le reste du dashboard
- **`.jf-ann-preview-modal` + `.pv-*`** : palette de classes pour le rendu live (head, title, body, compare table, footer, buttons)
- **Responsive 5 breakpoints** : mobile portrait (< 600 px), mobile/tablet (< 700 px / < 800 px), desktop large (≥ 1100 px)
- **Animation `jf-ann-pulse`** : keyframes pour le badge dirty

## 🔧 Détails techniques

- **`window.__md_saveAllAnnouncements`** : la fonction de save annonce est exposée sur `window` pour que le handler de save global puisse l'inclure dans `Promise.allSettled` sans dépendre du scope IIFE. C'est la seule pollution globale, justifiée par la nécessité de bridge entre les deux contextes
- **`readAnnouncementFromRow(row, existing)`** : nouvelle fonction pure qui lit une row de l'UI vers un objet `Announcement`. Réutilisée par le live preview ET le save. Garantit que ce que tu vois en preview = ce qui sera sauvegardé
- **Markdown rendering identique côté admin et côté banner.js** : `annMdRender` mirroir `mdToHtml` (bold, italic, lists, `[texte](url)` avec validation safe-scheme). L'aperçu admin est fidèle au rendu utilisateur final
- **Pas de nouvelle dépendance** : 100 % vanilla JS / CSS

## 🧪 Tests

- **121 tests xUnit** inchangés (Phase 1 de v0.3.9 inclut déjà 32 tests sur `AnnouncementHelper`)
- **Validation manuelle** : créer une annonce, vérifier que :
  - Les champs s'alignent proprement à toutes les tailles d'écran
  - Le live preview se met à jour à chaque keystroke
  - Le badge dirty pulse dès qu'un champ change
  - Le bouton *Enregistrer cette annonce* sauve isolément
  - Le bouton global *Enregistrer* sauve les annonces ET la maintenance
  - Les comparaisons s'ajoutent / suppriment proprement

## 📝 Notes

- Si tu avais une annonce créée en v0.3.9 et que tu mets à jour vers v0.3.10, **elle est conservée à l'identique** — ce sont uniquement les surfaces d'édition qui changent, pas le format de stockage
- Le viewport simulator dans l'admin (presets mobile / tablet / TV) reste prévu pour Phase 2. Pour tester un rendu sur un format spécifique, ouvrir Jellyfin sur ce device et activer une annonce pour ton compte uniquement (cible par UUID)
- Les modes carousel et stack restent en Phase 2 — toute annonce non vue est toujours rendue en mode `one-at-a-time` côté client
