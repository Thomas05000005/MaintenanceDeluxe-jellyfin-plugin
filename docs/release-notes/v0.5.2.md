# 🗓️ MaintenanceDeluxe v0.5.2 — Schedule sur annonces

Phase 2 annonces continue. Cette release ajoute le **système de planning** aux annonces — exactement le même que celui des `RotationMessage` (bannières rotatives) depuis v0.2.0.

Cinq types disponibles. La grosse différence avec les bannières : pour les annonces, le filtrage temporel se fait **côté serveur**, donc le client `banner.js` n'a aucune logique à comprendre — il reçoit la liste déjà filtrée.

## 🗓️ Les cinq types de schedule

### `always` (défaut)
Aucune contrainte temporelle. Comportement identique à v0.5.1 — c'est le défaut implicite pour toutes les annonces existantes.

### `fixed` — Fenêtre de date précise
Une seule plage `YYYY-MM-DD HH:MM` start/end. Idéal pour une annonce qui doit apparaître entre, disons, le 1er au 15 juin 2026.

```
Type: Fixed date range
Start: 2026-06-01 00:00
End:   2026-06-15 23:59
```

Bornes ouvertes acceptées : juste un `start` (annonce permanente à partir de X), juste un `end` (annonce jusqu'à X).

### `annual` — Récurrent par mois/jour
Pour les annonces saisonnières qui reviennent chaque année.

```
Type: Annual (recurring)
From: 12/20  To: 1/5      ← Christmas (wraps year)
Time: 00:00 - 23:59       ← optional time window
```

**Gère le wrap d'année** : `Dec 20 → Jan 5` est interprété correctement (active en décembre ET en janvier début). La logique est identique à `BannerSchedule` côté banner messages depuis v0.2.0.

### `weekly` — Jours de la semaine
Pour annonces récurrentes type "le mardi et jeudi".

```
Type: Weekly
Days: Mo, We, Fr
Time: 09:00 - 17:00       ← optional time window
```

### `daily` — Fenêtre horaire
La plus simple : "tous les jours de Xh à Yh".

```
Type: Daily
Time: 22:00 - 06:00       ← gère les fenêtres overnight
```

**Overnight windows supportés** : `22:00 - 06:00` est active de 22h à minuit ET de 0h à 6h. Logique identique au JS isInSchedule.

## 🏷️ Badges enrichis dans la liste admin

Quand une annonce a un schedule non-`always`, un **badge bleu supplémentaire** apparaît à gauche du badge d'état :

```
[ Annuel ]  [ Active ]            ← schedule annuel actif
[ Hebdo: Lu Ma Me ]  [ Active ]   ← weekly Mon/Tue/Wed
[ Quotidien 09:00-17:00 ]  [ Active ]   ← daily window
[ Fenêtre fixe ]  [ Brouillon ]   ← schedule fixed sur un brouillon
```

Le badge schedule s'affiche **EN PLUS** du badge état (Active / Inactive / Brouillon / Expirée), pas à la place. Tu vois en un coup d'œil : "ah, cette annonce annuelle est aussi en draft".

## 🔧 Sous le capot

### Backend C# — nouvelle helper testable

```csharp
internal static bool IsScheduleActive(BannerSchedule? schedule, DateTimeOffset now)
```

Pure, stateless, miroir exact du JS `isInSchedule` de `banner.js`. Pluggée dans `IsTargetedAtUser` (le serveur retourne déjà les annonces filtrées par schedule).

### Pourquoi serveur et pas client ?

Les bannières utilisent `isInSchedule` côté client parce que le serveur sert toute la liste de bannières et le client filtre selon la page courante (les bannières dépendent du route en plus du temps).

Les annonces utilisent `IsScheduleActive` côté serveur parce que :
1. Le serveur applique déjà les autres filtres (`IsActive`, `IsDraft`, `IsExpired`, roles, targetUserIds) ;
2. Pas de filtre route côté annonces (elles s'affichent toujours après login, peu importe la page) ;
3. Une annonce qui devient active à 14h n'a pas besoin d'attendre une nav SPA pour apparaître — la prochaine call à `/announcements/active` la livre.

### Validation au save

`SaveAdminAnnouncements` réutilise `ValidateSchedules` existant. Un admin envoyant `Schedule.Type = "monthly"` via DevTools reçoit un 400 BadRequest, pas un silent fallback.

### Admin UI

La fieldset `Planning` réutilise les helpers existants des rotation messages :
- `getScheduleDetailHtml()` — HTML statique
- `wireScheduleEditor(detail)` — events + boutons jour + presets annuels
- `populateSchedule(detail, schedule)` — chargement de l'état initial
- `collectSchedule(detail)` — sérialisation au save

Pas de duplication, pas de divergence : si on enrichit le composant un jour (ex. preset "Halloween" en plus du Christmas), les deux endroits en bénéficient.

## 🎁 Pour toi

Concrètement, tu peux maintenant :

- Préparer **une annonce "Soirée du Réveillon"** publiée le 1er décembre avec schedule `annual` Dec 20 → Jan 5. Tu ne touches plus jamais à cette annonce, elle revient toute seule.
- Mettre **une annonce "Mainte. mensuelle"** avec schedule `weekly` les premiers lundis (via combo schedule weekly + filtres serveur — note : "premier lundi du mois" n'est pas supporté directement, mais "tous les lundis" oui).
- Créer **une annonce d'astreinte** active uniquement la nuit avec daily 22:00 → 06:00.
- Combiner les filtres : une annonce **draft + schedule annuel Halloween** — invisible jusqu'à ce que tu décoches Brouillon, puis active uniquement entre le 25 octobre et le 1er novembre, chaque année.
