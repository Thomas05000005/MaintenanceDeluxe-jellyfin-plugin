# 🎨 MaintenanceDeluxe v0.6.0 — Éditeur de thème custom

**Phase 2 annonces complète.** Cette release boucle la promesse faite en v0.4.0 : tu peux maintenant créer ton propre thème d'annonce sans toucher au code. Palette, police, bordure — tout est pickable depuis l'admin.

## 🎨 Le nouvel éditeur

Dans l'onglet **Annonces**, un nouveau bloc dépliable **🎨 Éditeur de thème personnalisé** apparaît au-dessus de la liste des annonces. Sept champs :

| Champ | Format | Défaut Velours |
|---|---|---|
| **Nom** | Texte libre (max 40 chars) | "Custom" |
| **Couleur d'accent** | Hex `#RRGGBB` | `#C9A96E` (or) |
| **Backdrop** (fond derrière la carte) | Hex ou `rgba(...)` | `rgba(0,0,0,.68)` |
| **Fond carte** | Hex ou `rgba(...)` | Gradient brun-velours |
| **Couleur texte** | Hex | `#E6DBC9` |
| **Police** | Liste : Système / Inter / JetBrains Mono / Space Grotesk / Manrope | Système |
| **Bordure** | Liste : Solide / Halo lumineux / Pointillés / Aucune | Solide |

**Champs vides = défaut Velours.** Tu peux changer uniquement l'accent et garder tout le reste fidèle au Velours — pas besoin de tout configurer.

## 🟢 Activer le thème custom

Une fois le custom theme configuré, **"Personnalisé"** devient la 5ème option dans le segmented control du thème global, en haut de l'onglet :

```
[ Velours & Or ]  [ OLED Total Black ]  [ Glow Néon ]  [ Glassmorphism ]  [ Personnalisé ]
```

Tu peux aussi l'override par annonce (comme les 4 thèmes built-in). Et il fonctionne dans **les 3 modes** :
- **One-at-a-time** : ta modale s'affiche dans ton thème custom
- **Carousel** : chaque slide peut avoir son propre thème (les sliders peuvent mixer Velours + Custom + Neon)
- **Stack** : toutes les cards empilées partagent le thème de la première (cohérence visuelle)

## 🔥 Hot-reload dans la live preview

Quand tu changes un champ du custom theme, **l'aperçu admin se met à jour instantanément**. Pas besoin de sauvegarder. La CSS du thème custom est ré-injectée à chaque tweak via une signature JSON pour éviter le pile-up de tags `<style>`.

## 🔒 Sécurité

L'admin envoie du CSS — donc faille potentielle si pas validé. Trois lignes de défense :

1. **`NormaliseCssColor`** côté serveur — regex strict qui accepte uniquement `#RRGGBB` et `rgb()/rgba()` function notation. Rejette `red` (keyword), `hsl(...)`, `var(--x)`, et toute tentative de CSS injection (`rgba(...);position:fixed`).
2. **`NormaliseHexColor`** côté serveur pour accent/text (pas de transparence nécessaire).
3. **Whitelists** strictes pour `fontFamily` (5 valeurs) et `borderStyle` (4 valeurs). Tout ce qui n'est pas dans la liste est nullé.

Si un admin essaie de passer `rgba(0,0,0,.68);position:fixed;top:0`, ça finit en `null` au moment du save — le client n'a jamais l'occasion de rendre du CSS injecté.

## 🔧 Sous le capot

### Backend
- Nouvelle classe `CustomAnnouncementTheme` avec 7 champs nullable.
- Persistée dans `PluginConfiguration.CustomAnnouncementTheme`.
- `NormaliseCustomAnnouncementTheme(input)` valide chaque champ indépendamment et retourne `null` si tout est vide.
- `NormaliseCssColor(value)` : accepte hex et `rgb()/rgba()` avec regex strict.
- `POST /announcements/admin` reçoit un nouveau champ `customTheme` à la racine du payload.

### Client
- `/announcements/active` joint le `customTheme` à chaque item qui a `theme === "custom"` (économie réseau : items avec un autre thème n'ont pas le champ).
- `buildCustomThemeCss(t)` génère les règles CSS dynamiquement.
- `injectAnnTheme(themeKey, customTheme)` accepte le block et re-injecte au changement via signature JSON.
- `resolveAnnTheme` retourne `"custom"` tel quel (pas de fallback velours).

### Admin
- Nouveau bloc `<details>` repliable dans l'onglet Annonces.
- `collectCustomTheme()` extrait le DTO depuis les 7 inputs (null si tout est vide).
- `populateCustomThemeEditor()` charge la config au boot.
- Sauvegarde via le bouton Enregistrer global.
