# 🟠 MaintenanceDeluxe v0.5.1 — Brouillon + auto-expire

Suite directe de v0.5.0 (qui activait les modes carousel + stack). Cette release ajoute deux champs admin demandés depuis la Phase 1 : la possibilité de **préparer une annonce avant de la publier** (brouillon), et l'**expiration automatique** au bout de N jours.

## 🟠 Brouillon

Nouveau toggle dans l'éditeur d'annonce, séparé de `Active` :

```
┌─ Cycle de vie ─────────────────────────────────────┐
│  ☑ Brouillon (ne livre jamais, même si active)    │
│      Garde l'annonce visible en admin pour         │
│      préparer son contenu avant publication.       │
│                                                     │
│  Expire après (jours): [   ]  ← v.5.1 ↓            │
└─────────────────────────────────────────────────────┘
```

**Différence avec `Active`** :

| État | Active | Brouillon | Servie aux users ? | Visible en admin ? |
|---|---|---|---|---|
| Publié, en cours | ✅ | ❌ | ✅ Oui | ✅ Oui (badge vert) |
| En pause | ❌ | ❌ | ❌ Non | ✅ Oui (badge gris) |
| Préparation | ✅ ou ❌ | ✅ | ❌ Non | ✅ Oui (badge orange dashed) |

**Le brouillon trump tout** : peu importe `Active`, le serveur ne livre pas les drafts. Permet le workflow "j'écris l'annonce maintenant, je la publie demain" sans avoir à jongler entre actif/inactif (qui est sémantiquement "pause" et pas "préparation").

## ⏰ Auto-expire après N jours

Nouveau champ optionnel `Expire après (jours)` (1-365, vide = jamais).

**Cas d'usage** : annonces saisonnières que tu ne veux pas oublier de désactiver.

- Annonce "Soirée Halloween le 31 octobre" publiée le 25 octobre avec `expire=8` → disparaît automatiquement le 2 novembre.
- Annonce "Nouveaux films" publiée 1×/mois avec `expire=30` → garantit qu'aucune annonce périmée ne traîne.

**Comportement** :
- Filtrage **stateless** côté serveur dans `AnnouncementHelper.IsExpired`. Pas de mutation, pas d'auto-archive.
- L'annonce reste **visible en admin** avec un badge rouge barré "Expirée" pour comprendre pourquoi elle n'est plus servie.
- Si `PublishedAt` est `null` (annonce sans date), elle ne peut pas expirer (safe default).
- Valeurs invalides (0, négatif, > 365) sont clampées au save :
  - `0` ou négatif → `null` (= no expiration)
  - `> 365` → `365`

## 🎨 Badges enrichis dans la liste admin

La pilule à droite de chaque ligne annonce affiche maintenant 5 états :

| Badge | Couleur | Signification |
|---|---|---|
| `Active` | Vert | Livrée normalement |
| `Inactive` | Gris | Pause manuelle |
| `Brouillon` | Orange dashed | Préparation, pas livrée |
| `Expirée` | Rouge barré | Date d'expiration passée |
| `Active (3j)` | Orange | Expire dans ≤ 7 jours — heads-up visuel |

Priorité d'affichage : `draft` > `expired` > `inactive` > `active`. Si une annonce est draft ET expirée, le badge "Brouillon" s'affiche en priorité (le user ne verra de toute façon rien, le draft est plus actionnable côté admin).

## 🔧 Sous le capot

- **Backend** : 2 nouveaux champs dans `Announcement` (`IsDraft: bool`, `ExpireAfterDays: int?`). Filtres ajoutés dans `IsTargetedAtUser` et nouvelle helper `IsExpired(a, now)` pure et testable.
- **API** : `POST /announcements/admin` clamp `ExpireAfterDays` à 1-365. `IsDraft` est un bool simple.
- **Pas de migration** : les configs v0.5.0 chargent telles quelles, les nouveaux champs prennent leurs defaults (`false` / `null`) sur les annonces existantes.

## 🎁 Pour toi

- Tu peux maintenant **écrire l'annonce pour la sortie de v0.6.0 maintenant** et la garder en brouillon jusqu'au jour J. Pas besoin de jongler avec `Active` quand l'annonce n'est pas encore prête.
- Pour les **annonces saisonnières**, mets une expiration : tu n'as plus besoin de revenir manuellement les désactiver.
- Les **badges colorés** te disent en un coup d'œil pourquoi une annonce n'est pas servie (paused vs draft vs expirée).
- Si tu as une **liste qui grossit**, le badge orange "Active (Xj)" t'avertit visuellement quand une annonce va bientôt disparaître toute seule.
