# 🚀 MaintenanceDeluxe v0.3.12 — Fix : modal d'annonce après login

Bug remonté juste après l'install de v0.3.11 : tu crées une annonce active, tu te connectes en tant qu'utilisateur ciblé, **la modal n'apparaît jamais**. Cette release ferme le trou.

## ✨ Highlights

### 🐛 Le bug

`banner.js` exécute son code d'init **une seule fois** au chargement du script (injecté par JS Injector dans la page Jellyfin). Le flow était :

```
1. Fetch /maintenance (public, sans token)
2. install nav watchers (pour /maintenance)
3. Check token :
   - PAS de token (page login) → exit early, plus aucune action côté annonces
   - token présent → fetch /config + /User + tick() + fetchAndShowAnnouncements()
```

Le problème : si tu étais sur la page **login** (typique : navigateur qui démarre Jellyfin, ou session expirée), le script s'arrêtait au step 3 sans jamais déclencher les annonces. Et quand tu te connectais, Jellyfin **fait du SPA routing** : `pushState` vers `home.html`, pas de page reload. **`banner.js` n'était JAMAIS re-déclenché**, donc le hook annonces ne s'exécutait pas non plus.

Seul un user déjà loggé à l'ouverture initiale du browser pouvait voir une modal — pas le flow normal.

### 🔧 Le fix

Nouvelle fonction `maybeShowAnnouncements()` idempotente :

- **🔐 Track token** : stocke le token pour lequel on a déjà fait le check. Si même token → no-op. Si nouveau token (= login détecté) → trigger
- **🔄 Reset au logout** : si `getToken()` retourne `null`, on flush le tracking. La prochaine connexion re-déclenchera
- **⏰ Délai 800 ms** avant le pop du modal pour laisser le SPA home page se mounter visuellement (évite le modal qui apparaît avant la page elle-même)
- **🛡️ Skip si maintenance active** : pas de stack de modals plein écran

### 📡 Hookée sur chaque navigation SPA

Au lieu d'un seul appel au load initial :

- L'init initial appelle `maybeShowAnnouncements()` (déduplication = pas de double trigger pour l'utilisateur déjà loggé)
- `refetchAndApplyMaintenance()` (qui s'exécute à **chaque hashchange / popstate / viewshow**) appelle aussi `maybeShowAnnouncements()`
- **Résultat** : peu importe où l'user était à l'arrivée du script, dès qu'il se connecte et navigue, le hook fire correctement

### 🧹 Cleanup

- L'ancien `setTimeout(fetchAndShowAnnouncements, 1200)` directement dans l'init principal est remplacé par le simple appel à `maybeShowAnnouncements()` qui gère lui-même tous les cas (token, maintenance, idempotence, délai)
- Pas de double trigger possible — la garde `_announcementsCheckedForToken === tok` voit immédiatement qu'on a déjà checké ce token et early-return

## 🧪 Tests

- 121 tests xUnit inchangés (pas de logique business modifiée, juste le wiring du hook côté client)
- **Validation manuelle** :
  1. Crée une annonce active dans l'admin, cible-toi avec ton compte
  2. Déconnecte-toi de Jellyfin
  3. Reconnecte-toi
  4. La modal doit apparaître ~800 ms après l'arrivée sur la home page
  5. Dismiss → ne réapparaît plus pour cet user / ce session
  6. Reset 'vue par' depuis l'admin → reconnexion → la modal réapparaît

## 📝 Notes

- Si tu as déjà cliqué "Compris" sur une annonce et tu veux la re-voir pour tester : va dans l'admin, ouvre l'éditeur de l'annonce, clique **Réinitialiser 'vue par'**. Puis logout + login
- Le délai 800 ms est volontaire (UX) — la maintenance overlay garde son délai original car elle est plus prioritaire visuellement
- Aucun impact sur les autres surfaces (maintenance overlay, bannières rotation/permanent, admin UI). Touch chirurgical sur banner.js uniquement
