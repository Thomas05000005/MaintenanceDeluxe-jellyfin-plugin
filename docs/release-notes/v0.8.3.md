# 🛡️ MaintenanceDeluxe v0.8.3 — Durcissement qualité (réponse à un audit)

**Tu peux upgrader sans rien lire.** Pas de fix critique, pas de feature, pas de migration. Le DLL livré est **fonctionnellement identique à v0.8.2** : si v0.8.2 tourne bien chez toi, v0.8.3 fait exactement la même chose.

Alors pourquoi une release ? Parce que cette version pose **l'armure de tests et de CI** qui aurait attrapé le bug v0.8.0 (`MissingMethodException` sur Jellyfin 10.11.9) **avant** qu'il parte en prod. C'est une release pour le futur, pas pour le présent.

## Le contexte : deux fois le même genre de bug

- **v0.8.0** crashait sur 10.11.9 parce que Jellyfin avait supprimé `IUserManager.Users` au profit de `GetUsers()`. Invisible aux tests unitaires : ça ne pète qu'au runtime, quand un endpoint qui appelle `GetUsers()` est réellement invoqué sur le vrai serveur.
- **v0.8.1 / v0.8.2** ont corrigé puis re-vérifié la compat — mais à la main, par lecture de diff d'API.

La question de l'audit : *« qu'est-ce qui pourrait casser en prod que les tests actuels ne détecteraient PAS ? »* Réponse honnête : tout bug qui ne se manifeste qu'au chargement réel du plugin dans Jellyfin. v0.8.3 ferme ce trou.

## Ce qui change dans le code livré

Une seule chose, et elle ne change **rien** au comportement observable :

- **Extraction de `MaintenanceHelper.SetUserDisabledAsync`.** La glue Jellyfin (`GetUserDto` + `UpdatePolicyAsync`) était copiée-collée dans les 3 boucles `Activate` / `Deactivate` / drift-check — c'est précisément ce genre de glue API qui avait cassé sur 10.11.9. Désormais une seule implémentation, testée, qui dépend de délégués plutôt que de tout `IUserManager`. **Équivalence prouvée par mutation testing**, pas juste par relecture.

## Ce qui change dans le repo (mais pas dans ton zip)

Rien de tout ça ne part dans le plugin installé. C'est l'infrastructure qui garde le code honnête :

### 1. Un garde anti-« tests qui mentent » (Stryker.NET)

Le mutation testing modifie volontairement le code (inverse une condition, change un `>=` en `>`…) et vérifie qu'**au moins un test passe au rouge**. Si on peut casser le code sans casser un seul test, c'est que les tests n'assertent rien de réel. Ça tourne sur la logique pure (`AnnouncementHelper`, `MaintenanceHelper`), dans un workflow dédié (manuel + hebdo) avec un seuil de non-régression.

### 2. +21 tests (295 → 316), dont 15 tests d'intégration HTTP

Les endpoints d'admin (`SaveConfig` / `SaveMaintenance` / `SaveAdminAnnouncements`) sont maintenant pilotés à travers un **vrai** `Plugin` (le `BasePlugin` de Jellyfin, sur un dossier temporaire, avec un vrai `XmlSerializer`). On vérifie qu'ils rejettent bien les URLs `javascript:` et protocol-relative, clampent les valeurs hors bornes, assignent les IDs, purgent le tracking orphelin — **et que ça survit à un vrai aller-retour sérialisation → disque → désérialisation**. Construire le contrôleur contre les vraies assemblies Jellyfin 10.11.x est en soi une assertion de compatibilité à la compilation.

### 3. Un smoke test qui démarre un VRAI Jellyfin et bloque les releases ratées

C'est la pièce maîtresse. À chaque release, la CI :

1. construit le DLL,
2. le monte dans un **vrai conteneur Jellyfin** — en **matrix** sur `10.11.6` (le plancher du `targetAbi`) **et** `10.11.10` (ta prod),
3. complète le wizard de première install, s'authentifie en admin,
4. appelle `/MaintenanceDeluxe/users-summary` et `/MaintenanceDeluxe/announcements/admin` — **exactement** les endpoints qui appellent `IUserManager.GetUsers()`, la méthode qui crashait sur 10.11.9,
5. scanne les logs serveur pour toute `MissingMethodException` / erreur de chargement.

Si le plugin ne charge pas, ou si un endpoint pète, **la release est bloquée**. Le bug v0.8.0 n'aurait jamais atteint tes utilisateurs.
