# 🩹 MaintenanceDeluxe v0.6.1 — Hotfix install + audit logique

Cette release a deux objectifs : **débloquer l'installation Jellyfin de v0.6.0** (cassée par un typo dans `sourceUrl`) et appliquer les **fixes identifiés par l'audit logique** pré-release.

## 🔴 Le bug qui bloquait l'install

Quand tu essayais d'installer v0.6.0 via le manifest Jellyfin, tu voyais :

> **Une erreur s'est produite durant l'installation de l'extension.**

Cause : le `sourceUrl` dans `manifest.json` utilisait `maintenance-deluxe-jellyfin-plugin` en **minuscules**. GitHub URLs sont **case-sensitive** sur les paths des releases, donc :

```
https://github.com/Thomas05000005/maintenance-deluxe-jellyfin-plugin/...  → 404
https://github.com/Thomas05000005/MaintenanceDeluxe-jellyfin-plugin/...   → 302 OK
```

Jellyfin tentait de télécharger le zip, recevait 404, abandonnait l'install. Le hotfix initial est déjà sur main (commit `2a31e5f`) — cette release le formalise et corrige aussi les liens dans les release notes des v0.5.1 / v0.5.2 / v0.6.0.

## 🔧 Les 7 corrections de l'audit logique

### #1 — Custom theme invisible dans la preview admin

Quand l'admin choisissait "Personnalisé" dans le sélecteur global ou en override par annonce, **l'aperçu live affichait les styles velours par défaut**. Pourquoi ? Parce que `.theme-custom` n'existait pas dans `admin.css` — les 4 thèmes built-in avaient leur CSS de preview, le custom non.

Fix : nouvelle classe `.jf-ann-preview-modal.theme-custom` qui lit des variables CSS injectées inline par `renderAnnouncementPreview` :

```css
.jf-ann-preview-modal.theme-custom {
  background: var(--jf-custom-card, /* fallback velours */);
  color: var(--jf-custom-text, #E6DBC9);
  font-family: var(--jf-custom-font, inherit);
  border: var(--jf-custom-border, 1px solid var(--jf-ann-preview-accent));
}
```

L'admin voit maintenant son thème custom **en temps réel** dans l'aperçu, identique au rendu final côté users.

### #2 — ValidateSchedules ne vérifiait que le `type`

Cas concrets de bugs silencieux **avant v0.6.1** :

| Schedule envoyé | Comportement v0.6.0 | Comportement v0.6.1 |
|---|---|---|
| `{type:"fixed", fixedStart:"2026-06-01", fixedEnd:"2026-05-01"}` | accepté → annonce **jamais livrée** | **400** `fixedStart must be strictly before fixedEnd` |
| `{type:"annual", monthStart:13, ...}` | accepté → wrap inversé → **toujours active** | **400** `monthStart must be in 1..12` |
| `{type:"weekly", weekDays:[]}` | accepté → annonce **jamais livrée** | **400** `weekDays must contain at least one day` |
| `{type:"weekly", weekDays:[7]}` | accepté → annonce **jamais livrée** | **400** `weekDays must be in 0..6` |

L'admin sait maintenant **immédiatement** que sa config est invalide au lieu de constater plus tard que son annonce n'est jamais arrivée.

### #3 — `_pluginVersion` stale depuis v0.3.2

Le payload d'export config contenait `_pluginVersion: '0.3.2.0'` hardcodé en milieu de fichier. Pas mis à jour depuis 7 releases. Extrait en constante `PLUGIN_VERSION` au top d'`admin.js` — bumpée à chaque release comme le csproj.

### #4 — Sélecteur "Personnalisé" actif sans config

Avant : `_annData.customTheme === null` mais le radio "Personnalisé" reste cliquable. L'admin clique, rien ne change visiblement (fallback velours côté client), admin perplexe.

Après : option `disabled` + opacity 0.4 + tooltip explicatif. Auto-fallback velours si l'admin avait cliqué "Personnalisé" puis supprime sa config.

### #5 — Badges schedule incomplets trompeurs

Avant, le badge bleu `Fenêtre fixe` s'affichait même quand le schedule était... vide. L'admin pensait que tout était bon, mais l'annonce n'était jamais livrée (parce que `fixedStart` et `fixedEnd` étaient null).

Après : badge rouge dashed pour les configs incomplètes :
- `Fenêtre fixe (non configurée)` quand pas de start/end
- `Annuel (non configuré)` quand month/day pas tous renseignés
- `Hebdo (aucun jour)` reste mais avec style rouge

### #6 — `expireAfterDays` pas clampé client

Avant : admin tape `999` → input garde `999` → POST → serveur clamp à 365 → admin pense que `999` est persisté → confusion 6 mois plus tard quand son annonce expire mal.

Après : clamp côté client à la soumission + mise à jour de l'input. Admin voit `365` apparaître si jamais il a tapé une valeur trop grande.

### #7 — `NormaliseCssColor` 2 failles mineures

- `rgb(256, 0, 0)` était accepté (regex `\d{1,3}` matche 0-999) → CSS invalide, navigateur ignore, admin perplexe. Maintenant : bounds R/G/B in [0,255] enforced.
- `RGBA(...)` (majuscule) était rejeté alors que CSS est case-insensitive. Maintenant : regex `IgnoreCase`.

## 🔄 Refactor : `SAFE_URL_RE` centralisée

Depuis v0.4.1 (fix XSS protocol-relative), la regex `^(https?:\/\/[^\/]|\/(?!\/))/i` était dupliquée **4 fois** dans `banner.js` :
- ligne 510 (`safeUrl` dans showBanner)
- ligne 797 (`linkSafeUrl` dans mdToHtml)
- ligne 1848 (test ctaUrl)
- ligne 1945 (test imageUrl)

Identiques mais maintenance fragile : si on devait évoluer la regex, divergence garantie. Extrait en constante `SAFE_URL_RE` module-level. Les 4 sites utilisent maintenant la même référence.

## ✨ ANN_TEMPLATES enrichis pour Phase 2

3 templates quick-add modifiés/ajoutés pour exploiter les nouveaux champs Phase 2 :

- **🎬 Nouveaux films** : `expireAfterDays: 14` pré-rempli — les "nouveaux contenus" disparaissent automatiquement après 2 semaines.
- **🎃 Soirée Halloween** *(NEW)* : `schedule: { type: "annual", monthStart: 10, dayStart: 25, monthEnd: 11, dayEnd: 1 }` — revient chaque année automatiquement.
- **🎄 Fin d'année** *(NEW)* : `schedule: { type: "annual", monthStart: 12, dayStart: 20, monthEnd: 1, dayEnd: 5 }` — wraps year boundary correctement.
