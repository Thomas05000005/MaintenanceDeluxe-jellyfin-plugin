# 🖼️ MaintenanceDeluxe v0.5.3 — Image hero dans les annonces

Tu peux maintenant attacher une **image** (screenshot, poster, illustration) à chaque annonce. Elle apparaît au-dessus du corps de la modale.

## 🖼️ Le champ image

Deux nouveaux champs optionnels sur chaque annonce :

- **`imageUrl`** : URL de l'image. Mêmes règles que `ctaUrl` — `http://`, `https://`, ou path relatif `/...`. Le `//host` reste rejeté (voir v0.4.1 sécurité). Cap 2000 chars.
- **`imageAlt`** : texte alternatif pour l'accessibilité. Si vide, le client utilise le titre de l'annonce comme fallback. Cap 200 chars.

## 🎨 Rendu

Dans la modale, l'image apparaît **entre les métadonnées** (version + date) **et le corps** :

```
┌──────────────────────────────────────┐
│ 📣  Nouvelle UI déployée             │
│     v0.6.0 · 15 juin 2026            │
│                                      │
│  ┌────────────────────────────────┐  │
│  │                                │  │
│  │      [SCREENSHOT IMAGE]        │  │   ← v0.5.3
│  │                                │  │
│  └────────────────────────────────┘  │
│                                      │
│  Lorem ipsum dolor sit amet...       │
│                                      │
│                         [Compris]    │
└──────────────────────────────────────┘
```

`loading="lazy"` est activé : l'image n'est téléchargée que quand elle entre dans le viewport. Utile pour le mode `stack` quand plusieurs annonces avec image sont empilées.

## 🔒 Sécurité

Même allowlist que `ctaUrl`. La regex `^(https?:\/\/[^\/]|\/(?!\/))/i` (du fix XSS v0.4.1) est réutilisée :

- ✅ `https://example.com/screenshot.png` accepté
- ✅ `/MaintenanceDeluxe/assets/banner.jpg` accepté (chemin relatif, hébergé sur le même serveur)
- ❌ `//evil.com/track.gif` rejeté (protocol-relative, bypass d'host)
- ❌ `javascript:alert(1)` rejeté
- ❌ `data:image/svg+xml,<script>...</script>` rejeté (data: URIs)

Côté serveur, `SaveAdminAnnouncements` rejette toute URL invalide avec un **400 Bad Request** au moment du save. Le client ne reçoit jamais d'URL malveillante car la validation est effectuée avant la persistance.

## 🔧 Admin UI

Nouvelle fieldset **"Image (optionnelle)"** entre la fieldset "Corps" et "État & importance" :

```
┌─ Image (optionnelle) ──────────────────────────────┐
│  URL de l'image       Texte alternatif (a11y)      │
│  [                 ]  [                          ] │
│  ↳ https://... ou /   ↳ vide = utilise le titre   │
└─────────────────────────────────────────────────────┘
```

L'aperçu live (panneau droit de l'éditeur) **rend l'image en temps réel** dès que tu colles une URL valide.

## 🎁 Pour toi

- Une annonce **"Nouvelle UI"** avec un screenshot avant/après — plus parlant que du texte.
- Une annonce **"Soirée Halloween"** avec un poster d'événement — plus engageant.
- Une annonce **release** avec le logo de la nouvelle version — plus pro.
- L'image fonctionne dans **les 3 modes** : one-at-a-time, carousel (chaque slide a la sienne), et stack (cartes empilées avec leurs images respectives).

## 🧪 Tests

2 nouveaux tests xUnit :
- `Announcement_ImageUrl_DefaultsToNull` : vérifie que les nouveaux champs sont nullable et default à null
- `SelectDeliverableForUser_PreservesImageFields` : regression guard, l'image doit traverser la sélection serveur → client sans être perdue

**Total : 183 → 185 tests** (+2).

Pas besoin de plus : l'image est cosmétique, elle n'a pas de logique de filtrage propre. La validation IsUrlSafe est déjà testée exhaustivement dans `NormalisationTests` (depuis v0.4.1).
