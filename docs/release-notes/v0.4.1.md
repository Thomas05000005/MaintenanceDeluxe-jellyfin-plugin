# 🔒 MaintenanceDeluxe v0.4.1 — Audit de revue de code post-v0.4.0

Release **patch** sans nouvelle feature, focalisée sur la **qualité interne** : audit complet des ~10 000 lignes de code après le saut v0.4.0, **14 fixes** appliqués, **20 tests xUnit** ajoutés. Si t'as installé v0.4.0 et que tout marche bien, tu verras **zéro changement de comportement** côté admin. Mais sous le capot, une faille XSS exploitable a été bouchée, plusieurs fragilités d'UX corrigées, et `SelectDeliverableForUser` est passé en O(n) (de O(n²)).

## 🛡️ Le fix sécurité du milieu

Le regex qui validait les URLs dans les liens Markdown des annonces et des bannières acceptait sans broncher `[clique](//evil.com)`. C'est ce qu'on appelle une URL **protocol-relative** : un seul `/` en tête, suivi d'un domaine. Le navigateur la traite comme `https://evil.com` en relatif au protocole de la page. Résultat : un admin compromis (ou un compte de support malveillant) pouvait pousser une annonce qui, en un clic, envoyait tes utilisateurs vers un domaine pirate.

Le regex original :
```js
/^(https?:\/\/|\/)/i    // accepte //evil.com car ça commence par /
```
Le fix :
```js
/^(https?:\/\/[^\/]|\/(?!\/))/i    // le lookahead négatif refuse //
```

**6 endroits patchés** simultanément (3 dans `banner.js`, 2 dans `admin.js`, 1 dans `BannerController.cs`) pour garantir aucune désynchronisation entre client et serveur. Tests xUnit : `//evil.com`, `//evil.com/path`, `///triple-slash` tous rejetés.

## ⚠️ Modale de confirmation custom pour les actions destructives

Le `window.confirm()` natif est faible : un seul bouton OK qu'on peut cliquer accidentellement en deux tabs rapides au clavier. Remplacé par une vraie modale **`dangerConfirm`** sur trois actions :

- **Suppression d'annonce** — affiche le titre + le nombre d'utilisateurs qui l'ont déjà vue. Bouton danger désactivé pendant **3 secondes** avec compte à rebours visible.
- **Reset du suivi "vu par"** — affiche le titre de l'annonce concernée. Délai **2 secondes**.
- **Activation maintenance avec sessions actives** — affiche la liste exacte des users en train de streamer (nom + titre du film + appareil). Délai 1s, ou 3s si plus de 3 users actifs.

ESC pour annuler, Enter pour confirmer (si le délai est écoulé), clic hors carte pour annuler. Style dédié `.jf-dconf-*` avec backdrop blur.

## ⚡ Perf : `SelectDeliverableForUser` 50× plus rapide

Sur une instance avec 1000 users et 50 annonces, chaque appel à `/announcements/active` faisait ~50 000 comparaisons de strings (boucle `.Where().Any()` imbriquée). Maintenant : 1 `HashSet<string>` construit une fois pour le user concerné → ~1 000 comparaisons.

Pas visible sur petites instances. Sur les grosses, ça enlève une latence inutile à chaque navigation SPA.

## 🎨 Toutes les autres améliorations

### Sécurité défensive (jamais exploitable, mais propre)
- **Validation des couleurs hex** sur `ColorPresets`, `RotationMessages.bg|color`, `PermanentEntries.bg|color` dans `SaveConfig` : une valeur invalide (`red;position:fixed`) est silencieusement remplacée par le défaut du type au lieu d'être persistée. Nouvelle helper `NormaliseHexColorOrDefault(value, fallback)`.
- **`ValidateRoutes` hygiène** : rejet explicite des séquences `..` et `//` consécutives dans les patterns de route.

### Robustesse
- **Race condition token annonces** : `maybeShowAnnouncements` capture le token dans une closure, puis le **revalide** via `getToken()` après le délai 800ms. Si t'as logout pendant ce délai, le fetch programmé est abandonné au lieu de tirer avec l'ancien token.
- **Overlay annonce dédupliqué** : si une modale est encore en transition de sortie quand la suivante est déclenchée (mode "one-at-a-time", clic rapide), l'ancienne est explicitement retirée du DOM avant la création de la nouvelle.
- **`postMessage` admin→preview ciblé** : `window.location.origin` au lieu de `'*'`. L'iframe est same-origin, aucune raison de wildcarder.
- **Validation dates schedule** : `new Date('32/13/2025')` retourne `Invalid Date`, et `NaN <= NaN` est `false`. Avant : l'admin pouvait persister des dates invalides sans s'en rendre compte. Maintenant : `isNaN(d.getTime())` vérifié avant la comparaison.
- **`togglePanel` admin résilient** : itère sur `childNodes` pour trouver le premier text node au lieu de présumer `firstChild` (qui peut être une icône SVG selon le markup).

### UX / Perf
- **Throttle 30 fps sur `md-preview-update`** côté banner.js : un parent qui flood l'iframe (auto-save buggy, parent malveillant) ne peut plus pegger le CPU.
- **Debounce 120ms sur `updatePreview` admin** : le markdown n'est plus re-parsé à chaque keystroke (20×/s en saisie rapide → ~8×/s).

CI passe vert sur le `node --check`, ASCII-safety, version coherence, et les deux build steps Release.

## 🎁 Pour toi

Concrètement, après upgrade :

- **Tes annonces existantes** s'affichent exactement comme avant — aucun changement visuel.
- Si tu écris une nouvelle annonce avec un lien Markdown vers `//evil.com`, le lien sera supprimé silencieusement (seul le texte reste). Ça n'arrive jamais avec une URL légitime.
- Quand tu **supprimes une annonce** depuis l'admin, tu vois maintenant une vraie modale rouge avec le titre + compteur vu-par, et le bouton **Supprimer définitivement** est désactivé 3 secondes le temps que tu confirmes mentalement. Pareil pour Reset seen (2s) et activation maintenance avec sessions actives (1-3s).
- L'aperçu Markdown dans l'éditeur d'annonce est légèrement moins réactif (debounce 120ms) mais ça économise du CPU sur les vieux PC.
- Si tu as une grosse instance (centaines d'users + dizaines d'annonces), le endpoint `/announcements/active` est ~50× plus rapide.
